# 시스템 구조 설계

## 1. 전체 구조

```text
Client Console Program
        |
        | socket request / response
        v
Server Program
        |
        | file I/O
        v
users.txt / SMU_Students.txt
```

## 2. 클라이언트 책임

- 학생 관리자의 사용자 명령 입력
- 로그인 요청 송신
- 학생 검색, 추가, 삭제, 수정 요청 송신
- 학과명 기반 전체 학생 조회 요청 송신
- 신규 사용자 추가 요청 송신
- 사용자 삭제 요청 송신
- 서버 결과 출력

## 3. 서버 책임

- 학생정보 관리 초기화 작업 수행
- `SMU_Students.txt` 로딩 및 연결 리스트 구성
- 클라이언트 요청 처리
- 로그인 검증
- 학생 검색, 추가, 삭제, 수정 처리
- 학과명 기반 전체 학생 조회 처리
- 사용자 추가 및 삭제 처리
- 매일 01:00 학생정보 저장
- 서비스 제공

## 4. 파일 분리 구조

| 영역 | 파일 | 책임 |
|------|------|------|
| client | `client_main.c` | 클라이언트 실행, 서버 연결 |
| client | `client_ui.c` | 콘솔 메뉴, 입력, 결과 출력 |
| server | `server_main.c` | 서버 실행, 소켓 초기화, 연결 수락 |
| server | `request_handler.c` | 요청 메시지 해석과 명령 분기 |
| common | `protocol.h` | 메시지 포맷, 명령 코드, 공통 상수 |
| common | `student.c/h` | 학생 구조체와 연결 리스트 |
| common | `user.c/h` | 사용자 구조체와 사용자 파일 처리 |
| common | `file_io.c/h` | 파일 입출력 |
| common | `scheduler.c/h` | 매일 01:00 저장 시점 확인 |

## 5. 처리 흐름

1. 서버가 시작되면 학생정보 관리 초기화 작업을 수행한다.
2. 서버가 `SMU_Students.txt`를 읽어 학생 연결 리스트를 구성한다.
3. 클라이언트가 사용자 명령 기반 요청을 소켓으로 송신한다.
4. 서버가 요청 메시지의 명령을 판별한다.
5. 서버가 요청을 처리하고 응답 메시지를 반환한다.
6. 클라이언트가 서버 결과를 출력한다.
7. 서버는 매일 01:00에 학생정보를 `SMU_Students.txt`에 저장한다.
