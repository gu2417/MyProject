# 요구사항 분석서

## 1. 문서 개요

본 문서는 `2026학년도_1학기_중간고사대체과제.pdf`의 요구사항을 기반으로, **대학 교직원을 위한 온라인 학생관리 시스템**의 기능, 데이터, 제약사항, 예외 처리, 검증 기준을 상세히 정리한 요구사항 분석 문서이다.

AI 관련 주의사항은 본 분석 범위에서 제외한다. 단, 요구사항 외 기능 추가 금지, 지정 파일 사용, 도식 포함 등 시스템 및 산출물 품질과 직접 관련된 항목은 반영한다.

## 2. 시스템 목적

본 시스템은 대학 교직원이 학생 정보를 효율적으로 조회, 추가, 수정, 삭제할 수 있도록 지원하는 C 언어 기반 콘솔 프로그램이다. 시스템은 서버와 클라이언트로 분리되며, 사용자는 클라이언트 프로그램에서 로그인한 뒤 서버에 학생 정보 관리 요청을 전송한다. 서버는 학생 정보를 연결 리스트로 관리하고, 파일 기반으로 사용자 정보와 학생 정보를 유지한다.

## 6. 데이터 요구사항

### 6.1 학생 정보

학생 정보는 다음 항목을 포함해야 한다.

| 필드 | 설명 | 필수 여부 | 검증 기준 |
| --- | --- | --- | --- |
| 이름 | 학생 이름 | 필수 | 빈 값 불가 |
| 학번 | 학생 고유 식별자 | 필수 | 중복 불가 |
| 소속학과 | 학생이 소속된 학과명 | 필수 | 빈 값 불가 |
| 학년 | 학생 학년 | 필수 | 일반적으로 1~4 범위, 학교 정책에 따라 확장 가능 |
| GPA | 평균학점 | 필수 | 숫자 형식, 일반적으로 0.0~4.5 범위 |
| 이메일 연락처 | 학생 이메일 주소 | 필수 | 기본 이메일 형식 확인 |
| 모바일 폰 연락처 | 학생 휴대전화 번호 | 필수 | 빈 값 불가, 전화번호 형식 확인 |

### 6.2 사용자 정보

사용자 정보는 `users.txt` 파일에서 관리한다.

| 필드 | 설명 | 필수 여부 | 검증 기준 |
| --- | --- | --- | --- |
| 사용자 ID | 로그인 식별자 | 필수 | 중복 불가, 빈 값 불가 |
| 사용자 암호 | 로그인 암호 | 필수 | 빈 값 불가 |
| 사용자 소속부서 | 교직원 소속 부서 | 필수 | 빈 값 불가 |
| 사용자 이름 | 교직원 이름 | 필수 | 빈 값 불가 |

`users.txt`의 한 줄은 다음 형식을 따른다.

```text
사용자ID<TAB>사용자암호<TAB>사용자소속부서<TAB>사용자이름
```

즉, 한 사용자 레코드는 4개 필드와 3개의 탭 구분자로 구성된다.

### 6.3 학생 정보 파일

학생 정보는 `SMU_Students.txt` 파일에 저장한다. 파일 내 학생 세부 정보는 탭 문자로 구분한다.

권장 레코드 형식은 다음과 같다.

```text
학번<TAB>이름<TAB>소속학과<TAB>학년<TAB>GPA<TAB>이메일<TAB>모바일폰
```

서버는 프로그램 시작 시 `SMU_Students.txt`를 읽어 학생 연결 리스트를 초기화해야 한다. 서버는 학생 정보가 추가, 수정, 삭제된 후 변경 상태를 관리하고, 저장 시점에 파일로 반영해야 한다.

### 6.4 단과대학 정보

소속 단과대학을 표현하는 구조체 이름은 반드시 `DAEHWAK`으로 정의한다.

`DAEHWAK` 구조체는 단과대학 정보 기반 검색을 지원할 수 있어야 한다. 최소 포함 후보는 다음과 같다.

| 필드 | 설명 |
| --- | --- |
| 단과대학명 | 예: 공과대학, 인문사회대학 |
| 소속학과 목록 | 해당 단과대학에 포함되는 학과명 목록 |
| 소속학과 수 | 목록에 포함된 학과 개수 |

학생 정보에 단과대학명이 직접 저장되지 않는 경우, 시스템은 `DAEHWAK`의 학과 목록과 학생의 소속학과를 비교하여 단과대학 기준 검색을 수행해야 한다.

## 8. 기능 요구사항

### 8.1 서버 초기화

## 9. 서버-클라이언트 메시지 요구사항

계획서에는 서버와 클라이언트 간 원활한 요청 및 서비스 제공을 위한 메시지 포맷이 정의되어야 한다. 본 시스템의 메시지는 최소한 다음 정보를 포함해야 한다.

| 필드 | 설명 |
| --- | --- |
| 메시지 유형 | 요청 또는 응답 구분 |
| 명령 코드 | 로그인, 학생 추가, 학생 검색 등 요청 종류 |
| 인증 정보 | 로그인 이후 요청의 사용자 식별 또는 세션 정보 |
| 데이터 길이 | 전송 데이터의 길이 |
| 데이터 본문 | 실제 요청 또는 응답 데이터 |
| 결과 코드 | 성공, 실패, 인증 오류, 입력 오류 등 |
| 오류 메시지 | 실패 시 상세 사유 |

### 9.1 요청 메시지 예시

```text
REQ|LOGIN|0|user_id=kim;password=1234
REQ|STUDENT_SEARCH|session123|student_id=20260001
REQ|STUDENT_ADD|session123|student_id=20260002;name=홍길동;department=컴퓨터공학과;grade=2;gpa=4.1;email=a@b.com;mobile=010-0000-0000
REQ|DEPARTMENT_LIST|session123|department=컴퓨터공학과
REQ|COLLEGE_LIST|session123|college=공과대학
```

### 9.2 응답 메시지 예시

```text
RES|LOGIN|SUCCESS|session_id=session123
RES|STUDENT_SEARCH|SUCCESS|student_id=20260001;name=홍길동;department=컴퓨터공학과;grade=2;gpa=4.1;email=a@b.com;mobile=010-0000-0000
RES|STUDENT_ADD|FAIL|reason=duplicate_student_id
RES|DEPARTMENT_LIST|SUCCESS|count=35;data=...
RES|AUTH|FAIL|reason=login_required
```

구현 시 구분자 충돌을 방지하기 위해 입력값에 메시지 구분자가 포함되는 경우를 처리해야 한다. 단순 콘솔 과제 수준에서는 허용 문자 제한 정책을 둘 수 있다.

## 14. 핵심 구조체 후보

아래 구조체는 요구사항을 만족하기 위한 분석 단계의 후보이며, 실제 구현 시 C 문법과 파일 포맷에 맞게 구체화한다.

```c
typedef struct Student {
    char student_id[20];
    char name[50];
    char department[100];
    int grade;
    double gpa;
    char email[100];
    char mobile[30];
    struct Student *next;
} Student;

typedef struct User {
    char user_id[50];
    char password[100];
    char department[100];
    char name[50];
    struct User *next;
} User;

typedef struct DAEHWAK {
    char college_name[100];
    char departments[50][100];
    int department_count;
    struct DAEHWAK *next;
} DAEHWAK;
```
