# 요구사항 분석서

## 1. 문서 개요

본 문서는 대학 교직원을 위한 온라인 학생관리 시스템의 단일 최종 요구사항 문서이다.

기존 `requirements.md`의 본문 내용은 기준으로 사용하지 않으며, 본 문서의 양식, 제목 구조, 번호 체계, 표 형식, 요구사항 ID 중심의 작성 방식만 유지한다.

본 시스템은 Windows 운영체제를 사용하는 PC에서 정상 동작하는 C 언어 기반 콘솔 프로그램이다. 시스템은 학생정보를 관리하는 서버 프로그램과 학생 관리자가 사용하는 클라이언트 프로그램으로 분리한다. 서버와 클라이언트는 소켓 통신으로 요청과 응답을 주고받는다.

## 2. 시스템 목적

본 시스템은 대학 교직원이 학생 정보를 관리할 수 있도록 지원하는 C 언어 기반 Windows 콘솔 학생관리 시스템이다.

학생 관리자는 클라이언트 프로그램에서 로그인한 뒤 인가된 사용자로서 학생정보 관리 기능을 사용한다. 서버는 학생정보 관리 초기화 작업을 수행하고, 클라이언트의 학생정보 관리 요청을 처리하며, 학생정보 관리 서비스를 제공한다.

## 3. 사용자 및 이해관계자

| 구분 | 설명 |
| --- | --- |
| 개발요청자 | 대학 교직원을 위한 온라인 학생관리 시스템을 요구한 주체 |
| 학생 관리자 | 클라이언트 프로그램에 로그인한 뒤 학생정보 관리 기능을 사용하는 인가된 사용자 |
| 서버 프로그램 | 학생정보 관리 초기화, 클라이언트 요청 처리, 서비스 제공 담당 |
| 클라이언트 프로그램 | 사용자 명령 기반 요청 송신, 서버 결과 출력 담당 |

## 4. 시스템 범위

### 4.1 포함 범위

- Windows PC에서 동작하는 C 언어 콘솔 프로그램
- 학생정보를 관리하는 서버 프로그램
- 학생 관리자가 사용하는 클라이언트 프로그램
- 서버/클라이언트 소켓 통신
- 로그인 기능
- 로그인된 사용자만 학생정보 관리 기능 사용
- 로그인된 사용자만 신규 사용자 추가
- 사용자 삭제 기능
- `users.txt` 기반 사용자 정보 관리
- `SMU_Students.txt` 기반 학생 정보 관리
- 학생 검색
- 학생 추가
- 학생 삭제
- 학생 정보 수정
- 학과명 기반 전체 학생 조회
- 연결 리스트 기반 서버 학생정보 관리
- 파일 입출력 기반 데이터 로딩 및 저장
- 매일 01:00 학생정보 저장
- `time.h` 기반 시간 처리
- 사용자 편의성을 위한 메뉴 및 인터페이스
- 서버/클라이언트 송수신용 메시지 포맷 정의
- 서버 기능 구현용 구조체 및 구조체 멤버 정의
- 서버/클라이언트/공통 파일 분리 구조
- 헤더 파일 기반 구조체, 상수, 함수 원형 분리

### 4.2 제외 범위

최신 요구사항에 별도 제외 범위는 명시되지 않았다. 본 문서에 없는 기능은 추가하지 않는다.

## 6. 데이터 요구사항

### 6.1 학생 정보

학생 정보는 다음 필드를 포함해야 한다.

| 필드 | 설명 | 필수 여부 | 검증 기준 |
| --- | --- | --- | --- |
| name | 이름 | 필수 | 빈 값 불가 |
| student_id | 학번 | 필수 | 빈 값 불가, 중복 불가 |
| department | 소속학과 | 필수 | 빈 값 불가 |
| grade | 학년 | 필수 | 범위 기준은 요구사항 미정 |
| gpa | GPA(평균학점) | 필수 | 범위 기준은 요구사항 미정 |
| email | 이메일 연락처 | 필수 | 형식 기준은 요구사항 미정 |
| mobile_phone | 모바일 폰 연락처 | 필수 | 형식 기준은 요구사항 미정 |

### 6.2 사용자 정보

사용자 정보는 `users.txt` 파일에서 관리한다.

| 필드 | 설명 | 필수 여부 | 검증 기준 |
| --- | --- | --- | --- |
| user_id | 사용자ID | 필수 | 빈 값 불가, 중복 불가 |
| password | 사용자암호 | 필수 | 빈 값 불가 |
| department | 사용자 소속부서 | 필수 | 빈 값 불가 |
| name | 사용자 이름 | 필수 | 빈 값 불가 |

### 6.3 학생 정보 파일

학생 정보는 `SMU_Students.txt` 파일에서 관리한다. 학생 정보는 탭 문자로 필드를 구분한다.

| 순서 | 필드 | 설명 |
| --- | --- | --- |
| 1 | name | 이름 |
| 2 | student_id | 학번 |
| 3 | department | 소속학과 |
| 4 | grade | 학년 |
| 5 | gpa | GPA(평균학점) |
| 6 | email | 이메일 연락처 |
| 7 | mobile_phone | 모바일 폰 연락처 |

레코드 형식은 다음과 같다.

```text
name<TAB>student_id<TAB>department<TAB>grade<TAB>gpa<TAB>email<TAB>mobile_phone
```

### 6.4 사용자 정보 파일

사용자 정보는 `users.txt` 파일에서 관리한다. 사용자 정보는 탭 문자로 필드를 구분한다.

| 순서 | 필드 | 설명 |
| --- | --- | --- |
| 1 | user_id | 사용자ID |
| 2 | password | 사용자암호 |
| 3 | department | 사용자 소속부서 |
| 4 | name | 사용자 이름 |

레코드 형식은 다음과 같다.

```text
user_id<TAB>password<TAB>department<TAB>name
```

## 9. 서버-클라이언트 통신 요구사항

서버와 클라이언트는 소켓 통신으로 요청과 응답을 주고받아야 한다. 서버/클라이언트 송수신용 메시지 포맷을 정의해야 한다.

### 9.1 메시지 포맷

요청 및 응답 메시지는 탭 문자로 필드를 구분한다. 메시지 끝은 줄바꿈 문자로 구분한다.

```text
TYPE<TAB>COMMAND<TAB>STATUS<TAB>PAYLOAD
```

| 필드 | 설명 |
| --- | --- |
| TYPE | `REQ` 또는 `RES` |
| COMMAND | LOGIN, USER_ADD, USER_DELETE, STUDENT_SEARCH, STUDENT_ADD, STUDENT_DELETE, STUDENT_UPDATE, DEPT_LIST 등 명령 |
| STATUS | 요청 메시지는 `-`, 응답 메시지는 `OK` 또는 `FAIL` |
| PAYLOAD | 명령별 데이터. 여러 값은 탭으로 이어서 전송 |

### 9.2 요청 메시지

| ID | 요구사항 |
| --- | --- |
| NET-REQ-001 | 클라이언트는 사용자 명령을 요청 메시지로 구성해야 한다. |
| NET-REQ-002 | 요청 메시지는 소켓 통신으로 서버에 송신해야 한다. |
| NET-REQ-003 | 요청 메시지는 명령을 구분할 수 있어야 한다. |

### 9.3 응답 메시지

| ID | 요구사항 |
| --- | --- |
| NET-RES-001 | 서버는 요청 처리 결과를 응답 메시지로 구성해야 한다. |
| NET-RES-002 | 응답 메시지는 성공 또는 실패 상태를 포함해야 한다. |
| NET-RES-003 | 클라이언트는 응답 메시지를 수신하여 결과를 출력해야 한다. |

## 14. 필수 구조체 후보

서버 기능 구현을 위해 다음 구조체와 구조체 멤버를 정의한다. 배열 크기와 세부 상수값은 구현 시 헤더 파일에서 관리한다.

```c
typedef struct Student {
    char name[50];
    char student_id[20];
    char department[80];
    int grade;
    double gpa;
    char email[100];
    char mobile_phone[30];
    struct Student *next;
} Student;

typedef struct User {
    char user_id[40];
    char password[80];
    char department[80];
    char name[50];
} User;

typedef struct Message {
    char type[8];
    char command[32];
    char status[16];
    char payload[1024];
} Message;
```
